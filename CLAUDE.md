# CardForge: 중세 카드 경영 통합 실행 명세

`CLAUDE.md`의 제품 기준 문서와 `gemini.md`의 구현 전략 문서를 통합한 작업 기준서다.  
이 문서는 "무엇을 만들어야 하는가"와 "어떻게 안정적으로 구현할 것인가"를 동시에 제공한다.

## 프로젝트 목표
- Flutter 단일 코드베이스로 iOS / Android / Windows / macOS / Linux를 지원한다.
- 카드 드래그&드롭 조합을 중심으로 중세 생존/성장 경영 플레이를 구현한다.
- 게임 데이터는 로컬에만 저장하며, 서버/계정/외부 DB는 사용하지 않는다.

## 핵심 제품 규칙
- 입력은 `Draggable` / `DragTarget`으로 통합 처리한다.
- 레이아웃은 `LayoutBuilder` 기반의 반응형 비율 배치로 구현한다.
- 상태 관리는 Riverpod를 우선 사용한다.
- 저장은 `shared_preferences`만 사용한다.
- 플랫폼별 분기 코드는 최소화한다.

## 핵심 게임 로직
### 카드 타입
인류(사람, 전사), 건설(농지, 성벽, 배, 목조 가옥, 마을, 도시), 자연(바위, 나무, 토끼, 사슴), 위협(늑대, 곰), 재화(식량, 돌, 목재, 벽돌, 씨앗), 무기(창, 활, 궁수)

### 필수 조합 레시피
```text
돌 + 돌 -> 벽돌
벽돌 + 벽돌 -> 성벽
목재 + 돌 -> 창
목재 + 목재 -> 배
식량 + 식량 -> 씨앗
씨앗 + 농지 -> 과수원
창 + 창 -> 활
활 + 전사 -> 궁수
벽돌 + 나무 -> 목조 가옥
목조 가옥 + 목조 가옥 -> 마을
마을 + 마을 -> 도시
```

12개 이상으로 확장 가능해야 하며, 레시피 추가 시 UI 코드가 아니라 조합 테이블만 수정하도록 설계한다.

### 승리 조건
도시 1개 + 성벽 2개 + 인구(사람 + 전사) 10명 이상을 동시에 만족하면 승리.

### 패배 조건
사람 + 전사의 합이 0명이 되면 즉시 게임 오버. 생존 시간과 재시작 UI를 제공한다.

## 권장 아키텍처
### 1. 상태 구조
- `GameState`에 카드 목록, 자원 통계, 타이머 상태, 게임 진행 상태를 모은다.
- 카드 단위 모델은 `CardModel`로 분리하고 `id`, `type`, `position`, `timers`, `meta`를 가진다.
- 전체 보드 상태는 Riverpod `StateNotifier` 또는 `Notifier` 계열에서 관리한다.

### 2. 조합 시스템
- `CombinationEngine` 또는 `combinations.dart`에서 조합 규칙을 중앙 관리한다.
- `onAccept` 시점에 드래그된 카드와 대상 카드의 조합 가능 여부를 검사한다.
- 성공 시 기존 카드 제거, 결과 카드 생성, 이벤트 로그 기록까지 한 흐름으로 처리한다.
- 실패 시 시각적 피드백만 제공하고 상태를 훼손하지 않는다.

### 3. 게임 루프
- 개별 위젯 타이머 대신 중앙 집중식 Tick 기반 루프를 사용한다.
- 예: `Stream.periodic` 또는 타이머 기반으로 0.1초 간격 상태 업데이트.
- 이 루프에서 자연 스폰, 생존 타이머, 위협 처리, 승리/패배 감시를 일괄 처리한다.

### 4. 저장/복구
- `AppLifecycleState.paused` 또는 유사 시점에 자동 저장한다.
- 앱 시작 시 저장된 JSON을 복구해 이전 상태를 이어서 플레이할 수 있어야 한다.
- 로컬 저장 데이터는 체크섬 또는 간단한 무결성 검증 값을 포함해 비정상 수정 탐지를 고려한다.

## 권장 데이터 스키마
```json
{
  "last_saved": "2026-04-06T19:57:00",
  "stats": {
    "gold": 100,
    "pop": 12
  },
  "active_cards": [
    { "id": "uuid-1", "type": "CITY", "x": 0.5, "y": 0.3 },
    { "id": "uuid-2", "type": "WARRIOR", "x": 0.2, "y": 0.8, "timers": { "starve": 15.5 } }
  ],
  "best_records": {
    "survival_time": 1200
  },
  "checksum": "optional-integrity-hash"
}
```

## UI/UX 기준
- 화면 비율이 달라도 카드 배치가 깨지지 않아야 한다.
- 카드 겹침 또는 드롭 가능 상태에서 글로우 같은 시각 피드백을 제공한다.
- 모바일에서는 필요 시 햅틱 피드백을 연결할 수 있도록 구조를 열어둔다.
- 조합 성공, 자원 변화, 위협 발생, 승리/패배는 `event_log.dart`에 기록한다.
- 결과 화면은 승리 성취감 또는 패배 원인을 분명히 보여줘야 한다.

## 구현 우선순위
1. `CardModel`, `GameState`, 기본 카드 스폰 구조 정의
2. `CombinationEngine`과 조합 테이블 설계
3. 드래그&드롭 게임 보드 구현
4. 중앙 Tick 루프와 자연 스폰/타이머 처리 연결
5. 승리/패배 감시 및 결과 화면 연결
6. 자동 저장/복구 구현
7. 글로우, 이벤트 로그, 반응형 배치 등 UX 보강

## AI 작업 지침
- 새 기능을 추가할 때는 먼저 `게임 규칙 보존` 여부를 확인한다.
- 조합 규칙은 하드코딩 분기문보다 데이터 테이블 기반 확장을 우선한다.
- 상태 변경은 UI 위젯 내부보다 중앙 상태 로직에서 처리한다.
- 저장 구조를 변경할 때는 기존 저장 데이터와의 호환성 또는 마이그레이션을 고려한다.
- 성능 이슈가 생기면 개별 카드 위젯 rebuild 범위와 드래그 중 리렌더링 비용부터 점검한다.

## 핵심 구현 프롬프트
### 조합 시스템
"Flutter의 `Draggable`과 `DragTarget`을 활용해 12개 이상의 레시피를 처리하는 확장형 `CombinationEngine`을 설계하라. Riverpod 상태 관리와 연결하고, `onAccept`에서 조합 판정, 카드 제거/생성, 이벤트 로그 갱신까지 일관되게 처리하라."

### 반응형 UX
"`LayoutBuilder`를 활용해 모바일/데스크톱 공통의 반응형 카드 배치 시스템을 설계하라. 드롭 가능 상태에서는 글로우 피드백을 제공하고, 조합 성공/실패와 승리/패배 흐름을 사용자에게 명확하게 전달하라."

### 저장 및 게임 루프
"`shared_preferences` 기반 자동 저장/복구를 구현하라. 앱 일시정지 시 Riverpod 상태를 JSON으로 저장하고 재실행 시 복구하라. 또한 중앙 Tick 기반 게임 루프에서 자연 스폰, 타이머, 승리/패배 감시를 통합 관리하라."
